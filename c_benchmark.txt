# Benchmark plan: Rust `hqcr` vs. C reference implementations

## Context

We want a like-for-like running-time comparison between this pure-Rust HQC
library and the official C implementations in `c_implementations/`, across **all
three security levels** (HQC-128 / -192 / -256). Each level ships as two C
folders: a portable `reference-*` build and an AVX2+pclmul `optimized-*` build.
The Rust side already has criterion benches (`benches/bench.rs`, group `kem`:
keygen / encaps / decaps) that cover `hqc128`, `hqc192`, and `hqc256`. The C side
has **no timing harness** — `src/main_hqc.c` only runs one keypair->enc->dec
round-trip and prints the shared secrets. So the work is: build a small C speed
harness that mirrors the criterion `kem` benchmark, compile it into all six C
variants, run them, and collate the numbers against the Rust results.

Decisions already made with the user:
- **Scope: HQC-128, HQC-192, HQC-256.** All six folders now exist:
  - `reference-hqc-1`, `reference-hqc-3`, `reference-hqc-5` (portable `-O3`)
  - `optimized-hqc-1`, `optimized-hqc-3`, `optimized-hqc-5` (`-mavx -mavx2 -mpclmul`)
  Each is compile-time fixed to its param set in `parameters.h`. The matching
  Rust criterion numbers are `hqc128` / `hqc192` / `hqc256`.
- **Toolchain: MSYS2 mingw-w64 gcc** (`make` from MSYS2). Build commands target
  the **MSYS2 MINGW64 shell**.
- **Pairing (apples-to-apples), applied per level:**
  - C `reference-hqc-{1,3,5}` (portable)   <->  Rust **portable** build
  - C `optimized-hqc-{1,3,5}` (AVX2+pclmul) <->  Rust **`+pclmulqdq`** build

## Key findings (from exploration)

- C param set is compile-time fixed per folder in `parameters.h`. KEM byte sizes
  (from each `api.h`, `CRYPTO_*` macros):

  | Level | pk (PUBLICKEY) | sk (SECRETKEY) | ct (CIPHERTEXT) | ss (BYTES) |
  |-------|---------------:|---------------:|----------------:|-----------:|
  | HQC-1 (128) | 2241 | 2321 | 4433 | 32 |
  | HQC-3 (192) | 4514 | 4602 | 8978 | 32 |
  | HQC-5 (256) | 7237 | 7333 | 14421 | 32 |

- **The harness needs no per-level edits.** It sizes every buffer from the
  `CRYPTO_*` macros in `api.h`, so the same `main_speed.c` compiles correctly
  into all six variants and picks up that folder's param set automatically.
- No C compiler currently installed (`gcc`/`make`/`clang` absent); shell is MSYS2/MINGW64.
- The only non-portable dependency is `init_randomness()` in `main_hqc.c`
  (`syscall(SYS_getrandom,...)`). It is **not** used by any compiled object --
  `kem.c`/`symmetric.c` reference `getrandom` only in doc comments. The PRNG is
  the DRBG seeded by `prng_init(entropy, NULL, len, 0)`, so the harness can seed
  deterministically and skip the syscall. `prng_init`'s signature is identical
  across all six folders (`symmetric.h:71`).
- C KEM API (`api.h`, identical signatures in all folders):
  `crypto_kem_keypair(pk,sk)`, `crypto_kem_enc(ct,ss,pk)`, `crypto_kem_dec(ss,ct,sk)`.
- All six Makefiles are structurally identical; they differ only in the target
  name suffix (`hqc-1` / `hqc-3` / `hqc-5`) and in CFLAGS (reference is portable;
  optimized adds `-mavx -mavx2 -mpclmul`).
- Rust criterion (`benches/bench.rs`) times the **deterministic** variants
  (`keygen_from_seed`, `encaps_deterministic`, `decaps`) with fixed inputs and
  `black_box`, reporting wall-clock ns/op, for all three levels.

## Approach

### 1. New C speed harness -- `c_implementations/main_speed.c` (shared by all six folders)

Single file at the `c_implementations/` root, included by each Makefile via
`-I src` (resolves `api.h`/`parameters.h`/`symmetric.h` from whichever folder is
building). Mirrors the criterion `kem` group and is **level-agnostic**:

- All buffers sized from `CRYPTO_PUBLICKEYBYTES` / `CRYPTO_SECRETKEYBYTES` /
  `CRYPTO_CIPHERTEXTBYTES` / `CRYPTO_BYTES` — no hardcoded sizes.
- Print `CRYPTO_ALGNAME` at startup so output is self-labelling per level.
- Seed deterministically: `unsigned char entropy[48]; for i: entropy[i]=i;`
  then `prng_init(entropy, NULL, sizeof entropy, 0);` (same as the VERBOSE branch
  of `main_hqc.c`). **No `getrandom`, no `unistd.h`/`sys/` includes** -> builds on mingw.
- Iteration count from `argv[1]` (default 1000).
- **Correctness gate first:** one keypair->enc->dec, assert `ss1==ss2`, print
  `PASS/FAIL` before timing (confirms the build is sound).
- Time each op exactly as criterion does -- fix inputs outside the loop, measure
  the call repeatedly:
  - **keygen:** loop `crypto_kem_keypair(pk,sk)` into reused buffers.
  - **encaps:** fix `pk` once; loop `crypto_kem_enc(ct,ss,pk)`.
  - **decaps:** fix one `(sk,ct)` once; loop `crypto_kem_dec(ss,ct,sk)`.
- Timers: `clock_gettime(CLOCK_MONOTONIC,...)` for wall-clock ns (primary,
  comparable to criterion) **and** `__rdtsc()` from `<x86intrin.h>` for cycles
  (secondary). Record per-iteration samples.
- Report per op: **median**, mean, min ns/op and median cycles. Median ns/op is
  the headline number compared to criterion.
- **Anti-dead-code:** XOR a byte of every output (`ss`/`ct`/`pk`) into a
  `volatile` accumulator printed at the end, so `-O3 -flto` can't elide the calls.

### 2. Add a `hqc-{1,3,5}-speed` target to all six Makefiles

In each folder's Makefile, add a speed target mirroring that folder's existing
`hqc-N` target, swapping the main to the shared harness. The target name matches
the folder's level (`hqc-1-speed` in the `*-hqc-1` folders, `hqc-3-speed` in the
`*-hqc-3` folders, `hqc-5-speed` in the `*-hqc-5` folders):

    MAIN_SPEED:=../main_speed.c

    hqc-N-speed: $(HQC_OBJS) $(LIB_OBJS) | folders
    	$(CC) $(CFLAGS) $(MAIN_SPEED) $(addprefix $(BUILD)/, $^) $(INCLUDE) $(LIB) -o $(BIN)/$@

`CFLAGS` already differ correctly per folder (reference portable; optimized
`-mavx -mavx2 -mpclmul`), so the same harness compiles into all variants unchanged.

## Methodology caveats (to record in the doc)

- **Small RNG asymmetry:** C `crypto_kem_keypair`/`crypto_kem_enc` draw seeds /
  `m` / `salt` from the PRNG *inside* the timed call, whereas the Rust
  deterministic benches take those as fixed inputs. The difference is a couple of
  SHAKE squeezes -- negligible against the polynomial ops, but noted. (`decaps` has
  no such asymmetry.)
- **Cycle counter:** `__rdtsc` counts reference cycles (invariant TSC), not turbo
  core cycles -- indicative, not exact. ns/op is the authoritative metric.
- Run on the same machine, close background load, run 2-3x for stability; note CPU
  model and whether turbo/throttling is active.

## Verification

Run in the **MSYS2 MINGW64** shell (user runs all build/run commands -- never `cargo`/builds on their behalf).

One-time toolchain install:

    pacman -S --needed mingw-w64-x86_64-gcc make

C reference (portable), all three levels:

    cd c_implementations/reference-hqc-1 && make clean && make hqc-1-speed && ./bin/hqc-1-speed 2000
    cd ../reference-hqc-3 && make clean && make hqc-3-speed && ./bin/hqc-3-speed 2000
    cd ../reference-hqc-5 && make clean && make hqc-5-speed && ./bin/hqc-5-speed 2000

C optimized (AVX2+pclmul), all three levels:

    cd ../optimized-hqc-1 && make clean && make hqc-1-speed && ./bin/hqc-1-speed 2000
    cd ../optimized-hqc-3 && make clean && make hqc-3-speed && ./bin/hqc-3-speed 2000
    cd ../optimized-hqc-5 && make clean && make hqc-5-speed && ./bin/hqc-5-speed 2000

Rust (PowerShell, from repo root -- covers all three levels in one run each):

    cargo bench --bench bench -- kem
    $env:RUSTFLAGS="-C target-feature=+pclmulqdq"; cargo bench --bench bench -- kem; $env:RUSTFLAGS=""

Checks:
1. All six C binaries print their `CRYPTO_ALGNAME` and `PASS` on the round-trip
   gate before timing.
2. Each C harness emits median ns/op + cycles for keygen/encaps/decaps.
3. Plausibility cross-check against the known Rust decaps figures
   (from CLAUDE.md perf table); large divergence => investigate build flags:

   | Level | C opt decaps ~ Rust +pclmul | C ref decaps ~ Rust portable |

   | HQC-128 | ~0.87 ms | ~2.46 ms |
   | HQC-192 | ~2.22 ms | ~7.06 ms |
   | HQC-256 | ~4.73 ms | ~17.07 ms |
